# Performance Analysis Report: Wasm vs Native vs Docker

This report provides a detailed technical comparison of the `wasicell` runtime (WebAssembly/WASIX) against traditional deployment models.

## 🧪 Methodology
All tests were conducted on the `file-worker` service, which performs I/O operations (reading/writing files) to test both runtime overhead and filesystem virtualization costs.

- **Native**: Compiled for `x86_64-unknown-linux-gnu`.
- **Docker**: `debian:bookworm-slim` base image (~80MB).
- **Wasicell**: Compiled to `wasm32-wasmer-wasi` and executed via `wasmer-wasix`.

## 📈 Results Summary

### 1. Package & Image Size
| Format | Size | Comparison |
| :--- | :--- | :--- |
| **Wasm Module** | **1.07 MB** | **1.0x** |
| **Native Binary** | 4.33 MB | 4.0x larger |
| **Docker Image** | 79.4 MB | **74.2x larger** |

> **Finding**: WebAssembly provides the most compact distribution format, essential for edge computing where bandwidth and storage are constrained.

### 2. Cold Start Performance
Cold start measures the time from the execution command to the first line of guest code execution.

- **Native**: 0.80 ms
- **Wasicell (Wasm)**: **711.50 ms**
- **Docker**: 1,551.47 ms

> **Finding**: While native is nearly instantaneous, `wasicell` is **over 2x faster than Docker** for cold starts. This is a critical advantage for serverless and event-driven architectures.

### 3. Resource Isolation & Security
| Feature | Native | Docker | **Wasicell** |
| :--- | :--- | :--- | :--- |
| **Sandboxing** | None | Namespace/Cgroups | **Wasm Bytecode Sandbox** |
| **FS Access** | Full | Layered/Mounted | **Capability-based Virtual FS** |
| **Memory Safety** | OS-level | OS-level | **Memory-safe VM** |
| **Syscall Surface**| Full Linux | Restricted Linux | **WASIX (Minimal POSIX)** |

## 🧐 Technical Insights

### Why Wasm Wins at the Edge
1. **Startup Latency**: Unlike Docker, Wasm doesn't need to initialize a container rootfs or set up kernel namespaces. The runtime simply loads the module and begins execution.
2. **Security Model**: Capability-based security (WASI/WASIX) ensures that even if a guest is compromised, it cannot access the host network or filesystem unless explicitly granted permission in the `app.toml` manifest.
3. **Portability**: The same `.wasm` file runs on any OS/Architecture supported by Wasmer, removing the need for multi-arch Docker builds.
